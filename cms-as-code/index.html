üì¶ What is CMS as Code?

CMS as Code treats content models, schemas, and migrations as version-controlled code, bringing software engineering best practices to content management.

‚úÖ Benefits

  • Version control for content models
  • Code review for schema changes
  • Reproducible environments
  • Automated deployments
  • Rollback capabilities

üîÑ Workflow

  1. Define content types in code
  2. Commit to Git repository
  3. Review via Pull Requests
  4. Automate with CI/CD
  5. Deploy to environments
üí° Key Concept: Just like Infrastructure as Code (IaC), CMS as Code allows you to manage your content structure declaratively, track changes, and automate deployments.

üèóÔ∏è Infrastructure as Code for Content

Contentful spaces are like databases for content. Just as software requires infrastructure for each environment, content management systems need to be provisioned and configured.

üìã Provisioning

Scripts configure space environments with your content model

üîÑ Migration

Scripts migrate from older content models to current ones

‚úÖ Testing

Integration tests validate software works with current content model

üéØ Contentful provides: A composable content platform as a service which can be treated as code and included in your deployment pipeline. This requires provisioning scripts (to configure environments) and migration scripts (to update content models and data).

üì¶ Content Type Delivery

Similar to Drupal Configs: Content types are defined as code, versioned in Git, and deployed across environments.

üìù Content Type as Code

  • Define content types in migration files
  • Fields, validations, and settings in code
  • Version controlled in Git
  • Reviewable via Pull Requests

üöÄ Deployment Process

  • Create content type in dev
  • Commit migration to Git
  • Deploy to staging via CI/CD
  • Deploy to production after approval
üí° Key Concept: Just like Drupal stores content type configurations in YAML files that can be exported/imported, Contentful migrations define content types as code that can be versioned and deployed consistently across all environments.

üìö Content Type Versioning & Management

Content types are managed through sequential migration files, ensuring consistent delivery across environments.

üî¢ Sequential Migrations

  • Numbered migration files (1-, 2-, 3-...)
  • Each migration is a version
  • Run in order across environments
  • Tracked in version control

üîÑ Environment Sync

  • Same migrations run in all environments
  • Dev ‚Üí Staging ‚Üí Production
  • Consistent content model everywhere
  • No manual configuration needed

üìã Content Type Lifecycle

  • Create: Define new content type
  • Update: Modify fields or settings
  • Delete: Remove obsolete types
  • All tracked in Git history
üéØ Benefit: Content types are reproducible. Any environment can be recreated from migration files, just like Drupal can be rebuilt from config exports.

üîÑ Dual Lifecycle Challenge

Content management involves two simultaneous lifecycles:

üíª Software Deployment Pipeline

  • Developers author software versions
  • Code changes and releases
  • Uses dev, QA environments
  • Automated via CI/CD

üìù Content Creation Workflow

  • Content creators author content
  • Draft ‚Üí Review ‚Üí Publish
  • Uses production environment
  • Preview API for review
üí° Contentful Solution: Built-in workflow with "draft" and "published" stages, preview API for content review, and features to build advanced workflows. Content creation happens in production, while other environments are used for software deployment pipeline.

‚ö†Ô∏è Contentful Field Limitations

While Contentful is powerful, there is a significant limitation regarding field reusability.

üö´ Problem: Contentful does not support true field reusability across content types. You cannot define a field once and reuse it in multiple content types.

Impact:

  • You must duplicate field definitions for each content type
  • Changes to a "common" field require updating multiple content types
  • No centralized field definition or inheritance
  • Increased maintenance overhead
  • Higher risk of inconsistencies
üí° Example: If you need a "Title" field in BlogPost, Article, and Page content types, you must define it separately in each one. There's no way to define it once and reuse it.

üîÑ Migration Strategies: Expand-and-Contract

Contentful recommends forward-only migrations using the expand-and-contract pattern for safe, backward-compatible changes.

üìà Expand Phase

ADD - Don't Remove

  • Add new fields and content types
  • Transform content entries
  • Keep existing content in place
  • Support both old and new versions
  • Allow gradual rollout
  • Safe to rollback if needed

üìâ Contract Phase

REMOVE - After Success

  • Remove unneeded content types
  • Remove obsolete fields
  • Clean up old content
  • Run after successful rollout
  • Complete the migration
  • Final cleanup step
üí° Key Difference: Expand Phase adds new things while keeping old ones (safe, reversible). Contract Phase removes old things only after everything works (final cleanup).

üìä Expand Phase vs Contract Phase

Understanding when and why to use each phase:

Aspect üìà Expand Phase üìâ Contract Phase
Purpose Add new functionality safely Remove old, unused elements
Action ADD new fields/types DELETE old fields/types
Timing Before software deployment After successful rollout
Risk Low - nothing removed Higher - permanent deletion
Reversibility Easy to rollback Difficult (requires backup)
Old Content Kept intact Removed permanently
Software Compatibility Both old & new work Only new version works
üéØ Example: To change "author" from text to reference: Expand adds new "authorRef" field while keeping old "author" field. Contract removes old "author" field only after all systems use "authorRef".

ü§î Why Expand-and-Contract Pattern?

This pattern ensures safe migrations without breaking existing software.

‚úÖ Benefits

  • No downtime during migration
  • Old software continues working
  • New software can be deployed gradually
  • Easy rollback if problems occur
  • Safe to test in production

‚ùå Without This Pattern

  • Breaking changes immediately
  • Old software stops working
  • Requires coordinated deployment
  • Difficult to rollback
  • High risk of downtime
‚ö†Ô∏è Important: Contract Phase should only run after you're 100% sure the new version works everywhere. If in doubt, keep the old fields - they don't hurt anything.

Solution: Create helper functions for common fields

  • Define reusable field creators
  • Call them for each content type
  • Maintain consistency
  • Easier to update

Scenario: Migrate author from text to reference

  • Support both versions
  • Contract: Remove old field after rollout
üí° Key Points: Use helper functions to reduce duplication. Use expand-and-contract pattern for safe migrations that don't break existing software.
