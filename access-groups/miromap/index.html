Drag cards. Connections show how the app pieces work together.
Inspiration

Drupal Groups concept

  • Group-based editorial access
  • Consistent permissions + tagging
App Config

Access Group (UI model)

  • Access tag per group
  • Nested roles: Author / Editor
  • Permission matrix per content type
Compilation

permissionsSync → Contentful role policies

  • Generates JSON constraints
  • One-way sync to Contentful
  • Scoping is tag-driven
Safety

Deny tag editing (guardrail)

  • Prevents self-lockout scenarios
  • Deny rules injected from role spec
  • Tag membership stays authoritative
Runtime

Home: “Create Page” workflow

  • UI filters eligible content types
  • Backend re-checks create permission
  • Entry gets correct access-tag flow
Home

Needs Review + My Recent Edits

  • Needs Review: pending Tasks for the user
  • My Recent Edits: page entries updated by you
  • Both improve day-to-day navigation
Runtime

Sidebar auto-tagging (server-side)

  • Linked entries/assets get tagged
  • Uses server API: /api/apply-access-tag
  • Editors stay locked down safely
Business rule

Slug-pattern validation flow

  • Patterns configured per group
  • Hidden validation field blocks publish
Unification

Unified slug-validation + auto-tagging

  • Centralized existing module behavior into `access-groups`
  • Business adaptations: access-tag scoping + guardrails
  • Reduced separate maintenance/configuration overhead
Orchestration

Save app config → roles/tags/policies sync

  • Ensures “Access Group User” role
  • Creates group Editor/Author roles
  • Ensures tags exist, then syncs permissions
Transparency

Known limitations / risks

  • Tasks behavior may depend on token-owner behavior
  • UI filtering isn’t security; policies enforce access
  • Tag consistency is critical for scoping
